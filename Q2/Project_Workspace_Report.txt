=== 2. File Permissions Before and After Modification ===
--- project_plan.txt BEFORE chmod ---
-rw-r--r-- 1 aditya aditya 0 Jul 3 14:59 ProjectAlpha/docs/project_plan.txt
--- project_plan.txt AFTER chmod 700 ---
-rwx------ 1 aditya aditya 0 Jul  3 14:59 ProjectAlpha/docs/project_plan.txt
Explanation: The file was initially created with default permissions 644 (rw-r--r--) based on the system umask. After running chmod 700, permissions changed to rwx------, restricting all access to the owner only and removing read access for group and others.

--- deploy.sh AFTER chmod 750 ---

Explanation: deploy.sh was set to 750 (rwxr-x---), giving the owner full control while allowing group members to read and execute the script without being able to modify it - appropriate for a shared but access-controlled script.

=== 5. How Permissions Protect Project Data ===
Explanation: File permissions control who can read, write, or execute a file, limiting access based on ownership (owner/group/others). By setting stricter permissions like 700 on sensitive files (e.g., project_plan.txt), only the file owner can access it, preventing unauthorized users from viewing or editing confidential project data. Similarly, 750 on scripts allows trusted group members to run them without being able to modify the script, reducing the risk of accidental or malicious changes. This layered access control is fundamental to Linux security, ensuring that only authorized users can perform specific actions on files based on their role.
