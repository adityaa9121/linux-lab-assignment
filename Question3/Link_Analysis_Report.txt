=== 2. Comparison Between Hard Link and Symbolic Link ===
Hard Link (hardlink.txt): Same inode as original, link count increases, same size/permissions, breaks only if ALL hard links are removed, cannot cross filesystems, cannot link directories.
Symbolic Link (symlink.txt): Different inode, own small size (just the path string), permissions always 777 (lrwxrwxrwx), becomes 'dangling'/broken if target is removed, can cross filesystems, can link directories.
Explanation: The key difference is that hard links reference the same data on disk (same inode), making them resilient to the original filename being deleted, whereas symbolic links merely store a path to the target file and break if that path becomes invalid.

=== 5. Conclusions Drawn From the Experiment ===
Explanation: This experiment demonstrates that hard links and symbolic links behave fundamentally differently in Linux. Hard links are additional names for the same inode/data, making them robust against the original filename being deleted, but they cannot span filesystems or link directories. Symbolic links are independent files that store a path to a target, offering flexibility (can link directories, cross filesystems) but at the cost of breaking if the target is moved or deleted. Understanding this distinction is critical for data recovery scenarios, backup strategies, and safely managing files in a Linux environment.
