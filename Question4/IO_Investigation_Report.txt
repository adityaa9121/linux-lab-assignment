=== 1. Open Files (System-wide and Shell-specific) ===
lsof | head -20 showed system-wide open files, many owned by root (systemd, init, systemd-resolve) which returned 'Permission denied' since we are not root.
lsof -p $$ showed files open specifically in our shell process (PID 1838), including:
  - cwd: /home/aditya/lab-assignment/Question4 (current working directory)
  - txt: /usr/bin/bash (the shell binary itself)
  - mem: shared libraries like libc.so.6 and locale files loaded into memory
  - 0u, 1u, 2u: file descriptors for stdin, stdout, and stderr, all pointing to /dev/pts/0 (the terminal)
  - 7u: /dev/ptmx (pseudo-terminal master device)
Explanation: This shows that even a simple shell process has many files open at once - its executable, shared libraries, current directory, and the terminal device itself for input/output.

lrwx------ 1 aditya aditya 64 Jul  3 15:26 255 -> /dev/pts/0
lrwx------ 1 aditya aditya 64 Jul  3 15:26 7 -> /dev/ptmx
Explanation: This confirms visually that file descriptors 0 (stdin), 1 (stdout), and 2 (stderr) are symbolic links to /dev/pts/0, the terminal device. Descriptor 255 is bash's internal fd, also linked to the terminal, and fd 7 links to /dev/ptmx. This proves the Linux principle that 'everything is a file' - even keyboard input and screen output are represented as file descriptors pointing to device files.

=== 3. Output and Error Redirection ===
Command: echo "This is normal output" > output.txt
Result: stdout redirected into output.txt, containing: This is normal output

Command: ls nonexistentfile123 2> error.txt
Result: stderr redirected into error.txt, containing: ls: cannot access 'nonexistentfile123': No such file or directory

Command: echo "Both combined" > combined.txt 2>&1
Result: combined.txt contains: Both combined
Explanation: The > operator redirects standard output (fd 1) to a file, while 2> redirects standard error (fd 2) to a file. The 2>&1 syntax redirects stderr to wherever stdout is currently pointing, merging both streams into a single file - useful for logging both normal output and errors together.

=== 5. Explanation of How Linux Manages File I/O ===
Linux treats everything, including terminals, pipes, and devices, as files accessed through file descriptors (small integers starting at 0 for stdin, 1 for stdout, 2 for stderr). Each process maintains its own file descriptor table, visible under /proc/<pid>/fd, mapping these integers to actual open file objects in the kernel. Redirection (>, >>, 2>, 2>&1) simply changes which file descriptor points to which file or device, without the program itself needing any awareness of where its output is actually going. Resource limits (ulimit) cap how many such descriptors and other resources a process can allocate, protecting overall system stability. Together, this unified file abstraction and the file descriptor table mechanism are core to how Linux handles I/O, logging, and process communication.
